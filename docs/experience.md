本文件记录开发过程中已经踩过的坑、用户明确补充的偏好，以及会影响后续维护判断的实践经验。它不是替代 `docs/rules/development-guide.md` 的规则文档，而是解释规则背后的上下文，帮助后续修改少走回头路。

## 记录规则

- 当开发中发现可复用经验、重复问题、用户新增偏好、验证方式或架构取舍时，主动追加到本文。
- 经验条目应写清楚触发场景、经验结论和后续动作，避免只写口号。
- 已经升级为强约束的内容，应同步沉淀到 `docs/rules/development-guide.md`；本文保留背景和原因。
- 新增新的经验分类时，同步更新上方分类索引以及 `docs/README.md`。

## 前端样式与组件

### 2026-05-06 主题色需要通过 token 收敛

- 触发场景：主题色多次调整后，组件里残留旧色值和阴影色会让页面观感不一致。
- 经验结论：主题主色、hover、soft 背景、文字、图表线条、阴影和 focus ring 都应优先放在 `src/client/styles/base.css` 的 CSS 变量中，再由组件 CSS 引用。
- 后续动作：不要在组件 CSS 中散落品牌色硬编码；如果 `#85F2E3` 作为浅色主色用于按钮背景，文字和图表线条应使用同色系深色 token 保证可读性。

### 2026-05-06 组件拆分后必须同步保障样式入口

- 触发场景：业务组件拆分后，如果 CSS import 顺序或路径不完整，页面会退回近似裸 HTML 的状态。
- 经验结论：组件实现和同名 CSS 应放在同一个组件目录中，但 `src/client/styles.css` 仍必须作为统一入口维护所有样式 import。
- 后续动作：涉及 CSS 拆分或新增组件时，补静态资源测试，确认 `/assets/styles.css` 能引用并服务到组件样式。

### 2026-05-06 添加资产类型不是高频入口

- 触发场景：资产类型创建区域过大、位置过靠前时，会抢占月度记录这一高频任务的注意力。
- 经验结论：低频配置类动作应提供清晰入口，但不应该占用主工作区的主要视觉层级；桌面端应放在页面导航同一行的最右侧。
- 后续动作：后续布局调整优先围绕“记录月度价值”和“查看增值结果”组织，高频动作优先，配置动作弱化。

### 2026-05-06 资产历史抽屉需要更宽

- 触发场景：资产类型历史包含月度趋势、折线图和记录列表，窄抽屉会压缩信息密度，影响横向对比。
- 经验结论：桌面端抽屉宽度应保持在约 `780px`，也就是早期 `520px` 的 1.5 倍左右；移动端仍保持 `width: 100%`。
- 后续动作：后续修改 `components/ui/Sheet` 时，避免无意把桌面抽屉宽度调回过窄；如果不同场景需要不同宽度，应加明确 variant，而不是直接改全局默认。

### 2026-05-06 月度价值编辑只从明细抽屉进入

- 触发场景：把记录月度价值表单直接铺在主页面，会破坏主页面“查看汇总和明细”的扫描效率。
- 经验结论：更新资产数值应从月度明细行的“编辑/记录”按钮进入，在右侧抽屉中完成配置；主页面不展示独立的记录月度价值卡片。
- 后续动作：后续新增月度记录相关字段时，优先扩展 `RecordDrawer` 内的表单，不要把编辑表单重新放回主页面。

### 2026-05-08 总资产趋势贴近月度明细

- 触发场景：用户需要在月度明细上方查看总资产按月变化，方便先看整体趋势再下钻到资产类型明细。
- 经验结论：总资产折线图属于主 dashboard 的扫描信息，应放在汇总卡片之后、月度明细之前；不要藏进资产历史抽屉或操作记录页。
- 后续动作：后续扩展组合图、筛选范围或趋势指标时，继续保持它服务于“整体趋势 -> 明细下钻”的阅读顺序。

### 2026-05-09 资产趋势线条偏细

- 触发场景：趋势折线使用 4px 时视觉重量偏高，用户希望资产趋势线细一些。
- 经验结论：总资产趋势和资产历史趋势的折线宽度保持在约 2px，总资产趋势点位保持轻量，选中点半径约 4px。
- 后续动作：后续调整图表主题或重构趋势组件时，不要把折线默认宽度回调到粗线样式。

### 2026-05-09 月度历史需要显示最后更新时间

- 触发场景：资产历史抽屉只展示月份、价值和增值结果时，用户无法判断每个月记录最后一次维护的时间。
- 经验结论：资产历史表格应在每个月记录后展示最后更新时间，使用记录的 `updatedAt` 字段并保持弱化展示。
- 后续动作：后续调整历史表格列、移动端滚动或记录编辑流程时，保留最后更新时间的可见性。

### 2026-05-09 资产类型维护放在历史抽屉

- 触发场景：用户查看某个资产类型的月度变化时，需要顺手修改当前资产类型信息或删除该类型。
- 经验结论：资产类型编辑和删除入口放在资产历史抽屉头部；删除资产类型必须二次确认，并明确会级联删除其月度记录。
- 后续动作：后续扩展资产类型字段时优先复用历史抽屉的维护入口；如果要支持恢复资产类型删除，必须先设计级联记录恢复冲突策略。

### 2026-05-09 资产分组独立创建后再选择

- 触发场景：用户需要按分组查看资产统计，并明确希望先创建资产分组，再在资产类型里选择已有分组。
- 经验结论：分组应使用独立 `asset_groups` 表和创建入口；资产类型只保存可选 `group_id`，新增或编辑资产类型时用下拉选择分组。
- 后续动作：分组汇总必须沿用 `effectiveValue` 口径，本月未更新的资产类型也要进入所在分组；未分组用 `null` 表达，展示层显示为“未分组”。旧版 `group_name` 只作为迁移辅助字段，不应继续作为主写入字段。

### 2026-05-10 添加资产分组时展示已有分组

- 触发场景：用户在添加资产分组弹窗里需要看到现在已经有哪些分组，避免重复创建或离开弹窗核对。
- 经验结论：资产分组创建表单下方应直接展示已有分组列表和数量；列表来自当前 `assetGroups` 状态，创建成功刷新后立即更新。
- 后续动作：后续调整分组管理入口时，保留“创建 + 查看已有”的同场景反馈；如果增加编辑或删除分组，也应在这个列表附近扩展。

### 2026-05-10 Radix Select 在弹窗里需要更高层级

- 触发场景：用户在“添加资产类型”弹窗里点击分组下拉，只看到触发器聚焦，无法选择已有分组。
- 经验结论：`SelectContent` 使用 portal 挂到页面根节点，必须高于 Dialog overlay/content 的 `z-index`，否则下拉菜单会被弹窗遮罩挡住。
- 后续动作：新增或调整 Radix portal 类组件时，同步检查 `Dialog`、`Sheet`、`Select` 等浮层层级，避免菜单被父弹窗或遮罩盖住。

### 2026-05-06 明细数值列需要本地排序

- 触发场景：资产类型较多时，只看默认顺序不利于快速定位资产价值、增值金额或增值率最高/最低的项目。
- 经验结论：月度明细中当月价值、增值金额、增值率三列应支持本地升降序排序；空值始终排在最后，避免“待记录”打断有效数据比较。
- 后续动作：新增排序列时，把排序规则放在同组件目录的纯函数中并补单元测试，不要把复杂比较逻辑直接堆在 JSX 中。

## 数据与业务语义

### 2026-05-06 资产类型是持续跟踪维度

- 触发场景：早期实现容易把同一资产类型在不同月份当成多条独立记录创建。
- 经验结论：资产类型只创建一次，月度价值才是按月份变化的数据；月度视图应展示所有资产类型，并用状态表达当月是否已记录。
- 后续动作：不要通过重复创建资产类型或写入占位月度记录来让视图显示完整列表。

### 2026-05-09 总资产统计沿用最近记录

- 触发场景：某个资产类型本月没有更新时，如果总资产直接忽略该类型，会让总资产趋势出现不真实的断崖。
- 经验结论：指定月份汇总和总资产趋势应使用 `effectiveValue`：本月有记录用本月值，本月无记录则沿用目标月份之前最近一次记录；同时保留 `hasRecord: false` 和 `value: null` 表达本月未正式录入。
- 后续动作：后续修改汇总、趋势或排序逻辑时，要同时覆盖本月缺失、指定对比月份和趋势跨月份延续场景，不能写入占位记录。

### 2026-05-06 增值结果必须说明对比时间点

- 触发场景：只展示增值金额和增值率时，用户无法判断它们是相对上月、上一次记录，还是任意选定月份。
- 经验结论：增值金额、增值率和对比值都必须展示对比月份；汇总和明细应支持指定任意对比月份。
- 后续动作：新增汇总或图表展示时，把 comparison month 作为显式输入和输出，不要隐式藏在计算过程里。

### 2026-05-06 时间统一采用 GMT+8

- 触发场景：服务端或 SQLite 默认时间容易使用 UTC，导致操作记录和默认月份与用户预期不一致。
- 经验结论：前端默认月份、服务端创建/更新/恢复时间都统一按东八区生成。
- 后续动作：不要使用 SQLite `datetime('now')` 或运行环境本地时区作为业务时间来源。

### 2026-05-06 SQLite 迁移需要复制同名前缀文件

- 触发场景：迁移本地账本时，只复制 `assets.sqlite` 可能漏掉 WAL 模式下尚未 checkpoint 的写入内容。
- 经验结论：迁移或备份前先停止服务，然后复制数据库同名前缀文件，例如默认路径复制 `data/assets.sqlite*`，自定义 `ASSETS_DB_PATH` 时复制对应的 `my-assets.sqlite*`。
- 后续动作：文档或支持迁移场景时，明确包含主库、`-wal` 和 `-shm` 文件；不要要求迁移 `.omx/`、`node_modules/` 或其他无关目录。

## 测试与验证

### 2026-05-06 测试目标是暴露问题

- 触发场景：只写 happy path 会让删除恢复、冲突恢复、非法输入、空数据和边界月份问题漏掉。
- 经验结论：新增功能测试至少覆盖成功路径、失败路径、冲突场景和状态变化后的再次查询。
- 后续动作：涉及数据、接口、恢复能力或展示格式时，优先补能证明错误不会复发的测试，而不是只断言状态码。

### 2026-05-10 每个功能都要有完整测试 case

- 触发场景：用户明确要求“每个功能都要保证有完成的测试 case，不要都是 happy path”。
- 经验结论：后续新增或修改功能时，测试说明不能只写“已测成功路径”；至少要覆盖成功、校验失败、资源不存在或冲突、重复操作、取消/确认状态、状态变化后的再次查询等与功能风险匹配的场景。
- 后续动作：实现前先列出该功能的最小测试矩阵；如果某个场景暂时无法自动化测试，最终报告必须说明原因和剩余风险。

### 2026-05-06 前端视觉变化需要额外验证资源链路

- 触发场景：样式恢复和主题切换问题不一定能被类型检查发现。
- 经验结论：前端样式变动除了跑 `bun test` 和 `bun run typecheck`，还要确认 bundle 和 CSS 静态资源能被服务端正确返回。
- 后续动作：涉及入口、样式拆分或主题变量时，追加 `bun build src/client/main.tsx --target browser --outfile /tmp/assets-accretion-client-check.js`，必要时用临时端口和临时 SQLite 做 smoke。

### 2026-05-06 README 截图使用临时演示数据生成

- 触发场景：README 需要展示页面样式，但不能暴露或污染本地真实账本。
- 经验结论：页面截图应使用临时 SQLite 路径和演示数据生成，输出到 `docs/assets/dashboard-screenshot.png`。
- 后续动作：刷新截图时显式设置 `ASSETS_DB_PATH=/tmp/...`，结束后清理临时数据库和临时浏览器 profile，不要使用默认 `data/assets.sqlite`。

### 2026-05-10 测试 helper 必须防止写入正式账本

- 触发场景：拆分服务端和数据层测试后，多个测试文件会并行运行；如果共享全局 app/store 或直接使用默认 store，容易关闭彼此的数据库连接，甚至误写默认 `data/assets.sqlite`。
- 经验结论：每个测试文件应通过自己的 helper 上下文创建 app/store，数据库路径必须位于系统临时目录且文件名以 `assets-accretion-` 开头，并在 `afterEach` 清理主库、`-wal` 和 `-shm`。helper 里要用断言保护路径前缀，避免未来新增测试绕过临时库约束。
- 后续动作：新增 `tests/server/**` 或 `tests/server-db/**` 测试时复用现有 helper；不要直接调用 `createAssetStore()` 默认路径，也不要跨测试文件共享可变 app/store。

## 提交流程

### 2026-05-06 每次修改后自动提交

- 触发场景：用户要求代码修改完成后不要停留在未提交状态。
- 经验结论：完成修改、验证通过并确认暂存区只包含预期文件后，自动创建符合 Conventional Commits 的提交。
- 后续动作：除非用户明确要求暂不提交，否则每次修改后都提交；提交前仍要排除 ignored 的本地 SQLite、`.omx/` 和 `node_modules/`。
